## #3 masuda row 11 入力ミス未訂正（西出放置判断）

- **採番日**: 2026-04-19
- **採番起票**: α-B（v1.2.2 改訂時）
- **判断者**: 西出
- **対象レコード**: `mood_log_masuda` row 11 / `2026-04-15` evening (`T12:37:23`)

### 事実

- 増田は当該レコードを time_of_day=evening として記録したが、
  recorded_at が 12:37:23（昼）であり、本来 morning とすべき入力ミスと推定される（パターン 2c）。
- 西出は **訂正しない判断**を下した。

### 放置判断の理由

- 12:37:23 は morning_realtime_window `[06:00, 16:00)` 範囲内のため
  もし morning と訂正すれば entry_mode=realtime となり、原本の意図と乖離する可能性
- 増田本人の主観申告として `evening` が選ばれた事実そのものが観測値（§4.2 主観申告原則）
- 結果として entry_mode=retroactive のまま active で保持

### 設計書への反映

- §A.5.3 表に row 11 として追加（active / retroactive / 訂正なしで確定）
- §6.4「主観状態記録の原則」に整合する判断事例として記録

### 教訓

- Sheets 直接編集による訂正は**しない**判断も有効。
  入力時の主観申告そのものを観測対象として保持することは§6.4 原則に整合
- パターン 2c のすべてが訂正対象とは限らない。判断は西出（運用者）が行う

## #4 コード先行・設計書追従パターンの窓割り当て指針

- **採番日**: 2026-04-19
- **採番起票**: α-B（v1.2.2 改訂時）
- **対象**: 実装と設計書の整合管理ルール

### 事実

- v1.2 Sprint で以下のコード先行事例が頻発:
  - `daily_aspects` 列が Phase 1 時点で実装側により充填されていた
    （設計書は「Phase 2 で生成・Phase 1 は空文字」と記述・嘘状態）
  - α-B Phase 2 で `pending` 分岐を先行実装
    （設計書は v1.2.1 改訂で追従）
  - α-C Phase 2 で実データ手動追加対象を除外（NOT_RECORDED_EXCLUDE）
    （設計書は v1.2.2 改訂で追従）

### 窓割り当て指針

- **実装が先行している場合、設計書改訂は「実装に追従する」ことを基本方針とする**
- 改訂の窓割り当て:
  - α-B（実装担当）: 実装と設計書の差分を grep で検出 → 報告
  - α-A or 別 α: 改訂タスクとして起票・実行
- 改訂タイミング:
  - Sprint 完了時 / マイグレーションバッチ完了時 / 月次レビュー時 のいずれか
- 改訂後の検証: 実装が設計書記述を満たすかの逆向き grep を実施

### 補足

- 設計書を「コード生成の仕様書」ではなく「**運用実態と意思決定の記録**」として扱う方針
- 嘘状態（実装と設計書が乖離）を放置すると後続作業者が混乱するため、最優先で解消する

## #7 α-D GAS バッチ起動時の仕様申し送り

- **採番日**: 2026-04-19
- **採番起票**: α-B（v1.2.2 改訂時）
- **対象**: α-D（GAS バッチ実装担当）が着手前に確認すべき仕様前提

### α-D が確認すべき項目

#### (1) pending ユーザー扱い（§4.3.4 / v1.2.1 由来）

- realtime_window が settings.yaml で未定義のユーザーは `entry_mode=pending` を付与
- not_recorded 自動生成バッチは **pending ユーザーに対しては動作させない**
  （pending 自体が「判定保留」を意味するため、not_recorded との競合を避ける）
- suyasu の realtime_window 確定時に `devtools/rebalance_pending.py`
  （未実装）で pending → realtime/retroactive 再判定を実施する

#### (2) recorded_at 単桁時刻の lenient 化（mm_notes #1 由来）

- `2026-04-16T9:34:37` のような単桁時刻は ISO 8601 非準拠だが
  実データに混入し得る。GAS 側 parse 処理で lenient 化するか、
  事前正規化バッチで `T09:34:37` に置換するかを決定する
- Python 側は `migrate_v1_2._normalize_iso_jst` で lenient 化済（参考実装）

#### (3) record_id 再生成の責務境界（§A.3 v1.2.2 / mm_notes #2 由来）

- not_recorded 自動生成時の record_id は GAS 側で生成する
- アプリ層の訂正操作で recorded_at が変更された場合、
  record_id の同期再生成は**アプリ層の責務**（GAS バッチの責務ではない）
- GAS バッチは既存レコードの record_id を**触らない**

#### (4) BTR コホート GAS との分離（§A.7.6）

- スクリプトファイル分離方針（同 spreadsheet か別プロジェクトか）
- トリガー管理（time-driven trigger の重複防止）
- サービスアカウント権限の共有可否

#### (5) 実行時間上限（§A.7.6）

- GAS は 6 分制限。全ユーザー × 遡及期間（数十日 〜 数百日）の走査が
  収まるかの見積を事前に行う

#### (6) エラー通知（§A.7.6）

- Discord Webhook / メール / Sheets 内ログシート のいずれを採用するか

### 関連設計書箇所

- §4.3.2 not_recorded 生成トリガー
- §4.3.4 pending 判定
- §A.3 record_id 生成・再生成
